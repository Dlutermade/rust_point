# 02 — 點數帳本移動模型(退點・移轉・合併)

> 狀態:**草稿待審**(商業規格,不含 API 合約與資料表設計)
> 承接:`docs/plan/01-point-center/`(grant / redeem / expire 已定案)
> 現在式;繁體中文。

## 1. 目的與範圍

`plan/01` 定義點數核心:**發點**、**兌換(TCC)**、**到期**。本文補齊 **退點、移轉、合併**,並收斂出貫穿全部操作的統一模型。

## 2. 模型鐵律

### 鐵律一:批次不可變(append-only)

每一筆「加點」都是**發一列全新批次**,帶:

- **不可變**:`originalAmount`(發了多少)、生效期、到期日、**`transferable`(可否轉出)**。
- **可變**:`remainingAmount`(還剩多少)——純消耗計數,不是修改點數本體。

客戶餘額 = Σ 各批 `remainingAmount`。**發點、退點、移轉收入、合併收入都是發新批**;既有批次的條款**永不修改**(沒有「加回原批」「改效期」「改餘額」)。`transferable` 於加點時依政策設定(如**客戶賠償點常設不可轉出**),之後不可改。

### 鐵律二:帳本是回追鏈

帳本 `point_transactions` 嚴格 append-only,重點不在流水,而在**每筆都能追回它從哪來**:

- **消耗** → 記錄**吃了哪批**(消耗配置)。**兌換、移出、合併移出可跨多批**;只有到期是**單批**(一批到期一筆)。
- **加點(因他筆而生者)** → 記錄**因哪筆交易而生**(因果來源,單一指向)。

兩者串成鏈,可雙向回溯:`發點 → 被兌換消耗 → 觸發退點(新批)→ 又被消耗 → …`。

## 3. 操作全目錄

正號=發新批;負號=消耗。發起方決定權限:**前台發批的效期只能沿用來源,後台發批可自訂(留痕)**。

| 操作 | 帳本 | 發起方 | 新批效期 | 回追 | 分期 |
|------|------|--------|---------|------|------|
| grant 發點 | + 新批 | 系統/發點 | 發點時設定 | — | v1 ✅ |
| redeem 兌換 | − 消耗 | 前台客戶 | — | 吃了哪批(可多批) | v1 |
| release 釋放預留 | + 還原 | 系統 | — | 該次 redeem | v1 |
| expire 到期 | − 消耗 | 排程 | — | 哪批到期(單批) | v1 |
| **refund 退點** | + 新批 | 後台客服 | 客服設定 | 對應兌換(選填) | **v1** |
| transfer 移轉/贈點 | − 出 / + 入 | 前台客戶 | 沿用來源批 | 出:吃哪些批(多批);入:回追該移出 | v2 |
| merge 帳戶合併 | − 出 / + 入 | 後台客服 | 沿用來源批(客服可自訂) | 出:吃哪些批(多批);入:回追該移出 | v2 |

## 4. 名詞定義

| 名詞 | 定義 |
|------|------|
| 批次(point batch) | 一次加點形成的一列點數;`originalAmount` + 生效/到期 + `transferable`(可否轉出)**不可變**,`remainingAmount` 隨消耗遞減 |
| 消耗配置(allocation) | 一筆消耗記錄它扣了哪批、多少;兌換與移出可多批,只有到期單批 |
| 因果來源(origin) | 一筆加點所指向、導致它發生的另一筆流水帳(單一) |
| 來源溯源(provenance) | 每筆流水帳帶 `author`(誰發起)+ `sourceId`(對應外部單據);既是冪等鍵也是溯源 |
| 對手方(counterparty) | 移轉/合併中的另一位客戶 |

## 5. 操作規則

### 5.1 退點(refund)— v1

- **一件事,兩用途**:退單退還、客服補償(含 VIP、協商多給效期)本質相同——**後台客服發一批新點**,金額**任意**,效期由客服設定。
- **回追(選填)**:通常掛對應的兌換/訂單供溯源;非上限,也可不掛(純補償)。
- **已退可視**:後台可查「該兌換**已累計退多少**」(= Σ 回追到它的退點)與「原消費多少、當初吃了哪幾批」,供裁量參考;**是可視性,不是硬限制**。
- **可設不可轉出**:退點發批時可設 `transferable=false`(賠償/補償常見),防客戶把補償點轉贈他人。
- **性質**:立即完成、無 TCC、留痕;冪等鍵 `(author, sourceId)`(退貨單號 / 補償工單號)。

### 5.2 移轉 / 贈點(transfer)— v2

- 客戶 A 送點給 B,前台自助;**只能動 `transferable` 的批次**。
- **A 側(出)**:一筆 `transfer-out` 消耗,**可跨多批**(FIFO 挑可轉出批,記消耗配置);需足額可轉出、不透支。
- **B 側(入)**:依 `transfer-out` 消耗的**每個來源批**發一列對應新批,**各沿用該來源批的到期日**(防洗效期);每列都回追該筆 `transfer-out`。
- 冪等 `(author, sourceId)`。開放:是否需 B 同意 / 反詐欺額度(認證與風控 v2 一併談)。

### 5.3 帳戶合併(merge)— v2

- 客服把 A 帳號的點全數併入 B(同一人換帳號),後台操作。
- **A 側(出)**:`merge-out` 消耗 A **全部**有效批(多批);**不受 `transferable` 限制**(帳號整併非贈與他人,客服執行、留痕)。
- **B 側(入)**:依每個來源批發對應新批,沿用來源效期(客服可自訂,留痕),回追對應移出。
- 帳號身分不在本系統管理,本系統只搬點並留痕;冪等 `(author, sourceId)`。

## 6. 刻意排除:任意增減既有點數

**點數只透過兌換、移出、到期離開客戶;沒有倒扣、沒有改餘額、沒有改效期。**

- **加點自由(退點),減點不自由**——這個不對稱是刻意的:點數是商家對客戶的負債,樂於給、極少收。
- **改效期不存在**:「剩一天 → 多給 7 天」= 舊批照常過期,另發一批 7 天效期新點(就是退點)。
- **誤發**通常吃下當善意;真要處理是「那批的定向可回追作廢」(窄特例,另立規格),非任意倒扣。**詐欺**走帳號層級停權/凍結,不動點數餘額。

## 7. 稽核與分期

**稽核(未來)**:帳本是真相、`remainingAmount` 是 materialize 的可變狀態,故需唯讀對帳驗證無漂移——每批 `originalAmount − Σ消耗 == remainingAmount`;客戶餘額對帳本淨額;每筆退點/移入可沿回追追到成因。不一致發稽核事件。

**分期**:

- **v1**:核心 + **refund** + 帳本**回追來源**欄位。
- **v2**:transfer、merge、前台認證與風控。
- 但**模型現在就完整定義**,確保回追、對手方、效期規則一次到位,v2 不必回頭改結構。
